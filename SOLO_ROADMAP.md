# 单人 + Agent 驱动：研发任务时序（B）

> 你是架构师/产品经理/验收官；agent 是码农/算法/测试。
> 原则：先跑通最小闭环，再逐个替换 Mock。每个任务都写给 agent 的"范围+验收"，agent 交差你只做判卷。

---

## 阶段 A：骨架闭环（第 1–2 周，已交付脚手架）

**目标**：本地能跑通 `generate → 约束 → 对话 → 算量` 闭环。
**现状**：`home_agent_mvp/` 已可运行（pytest 6 passed，curl 验证通过）。

你这周只做：
- [ ] 跑通 `bash run.sh` + 两条 curl（已验证）
- [ ] 读 `app.py` / `agents/*.py` 理解数据流
- [ ] 给 agent 下阶段 B 任务（见 WEEK1_AGENT_PROMPTS.md）

---

## 阶段 B：替换 Mock，做真识别 + 真生成（第 3–8 周）

### B1 真规划 Agent（你拍板，agent 做）
- 任务：把 `planner.py` 占位替换为真实 LLM 调用（DeepSeek/Qwen API）。
- 验收：输入需求文本 → 输出结构化"房间功能+家具清单草案"JSON，可被 generator 消费。
- 你注意：LLM 只做推理，不生成最终尺寸数字。

### B2 真户型识别（agent 做，你验收）
- 任务：接开源方案（FloorSG / 或调用现成 API），替换 `understand.recognize` 的 Mock。
- 验收：清晰 CAD 图 → 结构化 house JSON，墙体≥85%、尺寸误差≤8%（MVP 放宽）。
- 你注意：承重墙判定沿用保守策略，不确定的标"疑似"。

### B3 真对话 Agent（agent 做）
- 任务：把 `dialog.py` 正则替换为 LLM Function Calling（edit_furniture / apply_style / move…）。
- 验收：口语指令 → 精准改 scheme JSON，覆盖尺寸/位置/风格/数量。
- 你注意：所有修改必须经 `constraint.evaluate` 校验，冲突返回替代方案。

### B4 构件库扩充（你主导品类，agent 录入）
- 任务：把构件库从 3 件扩到 30+ 件（床/柜/桌/沙发/厨卫），每件绑定 BOM 模板+单价。
- 验收：generate 能按需求拼出合理多件布局。

---

## 阶段 C：渲染 + 出图 + 上线（第 9–12 周）

### C1 真渲染（agent 做，你验收）
- 任务：前端接 Three.js 展示 scheme；或接商用云渲染试用。
- 验收：浏览器能看到 3D 户型 + 家具，可旋转。

### C2 工程出图（agent 做）
- 任务：把 `bom.py` 生产化，输出平面图 + 水电点位 + PDF 交付包。
- 验收：导出 PDF 含效果+尺寸+图纸+报价，工长能看懂。

### C3 小程序/H5 上线（agent 做）
- 任务：微信小程序或 H5 包一层，接上述 API。
- 验收：真实用户能上传户型图 → 说话 → 拿方案。

### C4 真实用户测试（你做）
- 任务：找 5–10 个真实装修小白测，记录卡点。
- 你注意：这是唯一不能外包的环节——你亲自看用户怎么用。

---

## 哪些事不能交给 agent（你必须亲自）

1. **承重墙/安全红线判定的最终口径**（agent 可建议，你定规则）。
2. **真实用户测试**（agent 测不出"小白卡在哪"）。
3. **产品方向决策**（做什么不做什么）。
4. **对外承诺/合规文案**（责任在你）。
