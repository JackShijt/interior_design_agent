# 0. 文档摘要（Executive Summary）

## 0.1 一句话定位
**"上传户型图，用说话和发图，一键生成可施工的室内设计方案（3D 效果 + 精确尺寸 + 施工图 + 报价），并持续对话优化。"**

## 0.2 要解决的问题
当前室内设计软件（酷家乐、三维家、Homestyler 等）本质是**给人用的工具**，而非**替人服务的 Agent**：
- 小白面对空场景无从下手，仍需"学软件 + 自己布局"。
- "AI 装饰"多为套模板换风格，不理解户型的结构约束（承重墙、管井、净高）。
- 3D 方案、施工图、尺寸、报价是**割裂的三条线**，无法一键联动。
- 缺乏"改一句话 → 几何/渲染/图纸/报价同步更新"的闭环。

## 0.3 核心价值主张
把"设计师的认知能力"压缩成一个 AI Agent：
1. **约束理解**：自动识别承重墙、管线、尺寸，遵守规范与人体工学。
2. **一键生成**：从户型图直接产出可交付方案包（非仅效果图）。
3. **自然语言 + 参考图交互**：小白用说话/发图推进，无需菜单。
4. **全链路同步**：任何修改实时重算几何、渲染、施工图、报价。

## 0.4 技术可行性判断
所有单点技术（户型识别 CV、生成式 3D、云渲染、LLM 推理、BIM 算量）均已成熟。
**真正的创新与壁垒不在单个算法，而在于"约束感知 + 全链路数据骨架 + 对话闭环"的系统集成。**

# 1. 产品愿景与用户场景

## 1.1 产品愿景
成为"每个人的专属室内设计顾问"——零门槛、可交付、可对话。

## 1.2 目标用户
| 用户类型 | 核心诉求 | 当前痛点 |
|----------|----------|----------|
| 装修业主（小白） | 快速看到自家效果、知道花多少钱、怎么施工 | 看不懂图纸、怕被设计师/工长坑 |
| 租房/二手房改造者 | 低成本、可逆的轻改造方案 | 不想大动、需快速出图 |
| 软装用户 | 参考图落地到自己家 | 喜欢图但不知如何适配自家尺寸 |
| 小微装修公司 | 快速出方案谈单、自动报价 | 设计师人力贵、出图慢 |
| 电商/家居品牌 | 方案即导购 | 用户难想象实物摆自家效果 |

## 1.3 关键用户旅程（小白视角）
```
[上传户型图] → 自动识别结构
      ↓
[说一句："我想要现代简约，两口之家，多收纳"]
      ↓
[10 秒内生成]
   ├─ 3D 效果图（可 720° 看）
   ├─ 精确尺寸标注（每面墙、每件家具）
   ├─ 施工图（CAD 格式，可给工长）
   └─ 物料清单 + 报价
      ↓
[对话优化]
   "主卧衣柜加长 30cm"
   "参考这张图换个暖色调"
   "厨房做成开放式"
      ↓
[全链路实时重算并刷新]
```

## 2.2 设计第一原则：单一数据骨架（Single Source of Truth）
**所有下游（3D 渲染、施工图、报价、对话修改）都由同一份"结构化方案 JSON"驱动。**
这是全链路同步的技术基石——改 JSON 一处，所有视图重算。

```json
{
  "project_id": "...",
  "unit": "mm",
  "house": {
    "walls": [{"id":"w1","p1":[0,0],"p2":[4200,0],"thickness":200,
               "type":"bearing|partition","height":2800}],
    "openings": [{"id":"d1","wall_id":"w1","type":"door|window",
                  "offset":1200,"width":900,"height":2100}],
    "rooms": [{"id":"r1","name":"主卧","type":"bedroom",
               "poly":[[0,0],[4200,0],[4200,3600],[0,3600]],
               "area":15.12}],
    "fixed": [{"type":"pipe_shaft","poly":[...]},{"type":"beam","z":2600}]
  },
  "design": {
    "style": "modern_minimal",
    "furniture": [{"id":"f1","cat":"wardrobe","model":"wd_01",
                   "pos":[100,200],"size":[1800,600,2400],
                   "color":"#E8E2D8","price":3200}],
    "harddecor": [{"type":"floor","material":"oak","room":"r1"}]
  },
  "engineering": {
    "bom": [...], "quotation": {...}, "drawings": [...]
  }
}
```

# 3. 核心模块详细设计

## 3.1 理解 Agent：AI 户型识别（输入解析）

### 3.1.1 功能
将用户上传的**户型图照片 / 扫描件 / CAD** 转换为结构化户型 JSON（含真实毫米尺寸、承重墙标记、门窗、房间语义）。

### 3.1.2 技术实现（基于 CVPR 2021 住宅户型识别与重建升级）
```
输入图像
  → ① 图像预处理（去噪/透视校正/对比增强）
  → ② 语义分割网络（CNN/Transformer）：墙/门/窗/承重墙标记/房间
  → ③ 目标检测：门窗符号、家具图标、管井/烟道
  → ④ OCR：尺寸标注数字（"4200"）+ 房间名（"主卧"）
  → ⑤ 墙线矢量化：骨架提取→霍夫变换→端点/拐点坐标→拓扑闭合
  → ⑥ 比例校准：用已知标注尺寸反推像素→毫米比例
  → ⑦ 微分渲染闭环：把几何渲染回图比对，反向优化参数精度
  → ⑧ 人工校正 UI（可选）：用户拖动修正，存为训练反馈
```

### 3.1.3 关键难点与对策
| 难点 | 对策 |
|------|------|
| 拍照畸变/阴影 | 透视校正 + 多图融合 |
| 手绘/CAD 线稿风格差异 | 多域训练数据 + 风格归一化 |
| 尺寸缺失/矛盾 | 规则补齐（标准开间模数）+ 用户确认 |
| 承重墙误判（高风险） | 保守策略：标记"疑似承重"并提示，禁止 AI 擅自拆改 |
| 老房图纸不规范 |  fallback 到"手动描图"辅助模式 |

### 3.1.4 输出
结构化 `house` 对象（见 2.2 JSON），作为全系统唯一空间真相源。

## 3.2 约束引擎：可施工性的守门员（核心壁垒）

### 3.2.1 为什么必须有约束引擎
生成式 AI 会"自由发挥"，但室内设计有**硬边界**：
- 承重墙不可拆、管井不可动、梁下净高限制；
- 规范：疏散、插座高度、防水区域；
- 人体工学：通道≥900mm、衣柜深度 550–600mm；
- 工艺可行性：瓷砖对缝、吊顶厚度。

**没有约束引擎，AI 生成的只是"好看但没法施工"的图。**

### 3.2.2 约束规则库结构
```
规则分类：
  A. 结构安全（红线，不可违反）
     - 承重墙禁止拆除/开大洞
     - 管井/烟道禁止移位
     - 阳台荷载限制
  B. 规范合规（强提示，需确认）
     - 消防/疏散通道
     - 卫生间防水范围
     - 插座/开关高度
  C. 人体工学（软约束，优化目标）
     - 动线宽度、家具间距
     - 收纳面积占比
  D. 工艺可行（生成时校验）
     - 材料拼贴模数、收口
```

### 3.2.3 执行方式
- 每条规则可形式化为：`IF 条件 THEN 动作(阻断/警告/优化)`。
- 在**生成前**（裁剪不合规方案）、**生成中**（约束优化）、**生成后**（校验报告）三阶段介入。
- 输出"合规性报告"：哪些满足、哪些需人工确认、风险提示。

## 3.3 规划 Agent：空间推理（LLM 驱动）

### 3.3.1 角色
把用户自然语言需求（"两口之家、多收纳、现代简约"）翻译成**空间分配策略**：
- 哪些房间做什么功能、面积如何分配；
- 动线如何组织；
- 收纳系统优先级。

### 3.3.2 实现
- 用 LLM（带领域 Prompt + Few-shot）做"空间规划推理"。
- 输入：户型 JSON + 用户需求 + 风格向量。
- 输出：房间功能分配 + 家具清单草案（类别/数量/大致位置），作为生成 Agent 的约束输入。
- 例：`主卧→床+衣柜+床头柜；衣柜靠东墙（避梁）；收纳目标≥3㎡`。

### 3.3.3 价值
让"小白说的话"变成"机器可执行的布局指令"， bridging 自然语言和几何。

---

## 3.4 生成 Agent：约束感知的方案生成

### 3.4.1 两条生成路线（建议并行）

**路线 A：可控构件拼装（MVP 首选，保证可施工）**
- 从参数化构件库选取家具/硬装，按规划 Agent 策略 + 约束引擎摆放。
- 优点：100% 可施工、尺寸精确、直接出 BOM。
- 实现：几何约束求解（靠墙、对齐、间距）+ 规则布局算法。

**路线 B：生成式 3D（进阶，提升惊艳度）**
- 文本/参考图 → 扩散/3DGS 生成视觉方案 → 反投影到构件库做"贴合"。
- 优点：风格自由、参考图还原度高。
- 风险：需后处理保证可施工，MVP 后引入。

### 3.4.2 参考图风格提取
- 用户发参考图 → CLIP / 扩散模型提取风格向量（色调、材质、造型语言）。
- 应用到自家户型的约束下（不照搬布局，只借鉴风格）。
- 解决"我喜欢这张图，但怎么落在我家"的痛点。

### 3.4.3 输出
填充 `design` 对象（家具/硬装/材质/尺寸），驱动下游。

---

## 3.5 渲染层：云渲染引擎（类 KooEngine）

### 3.5.1 功能
将 `design` JSON 实时渲染为：
- 3D 效果图（照片级，光追）
- 720° 全景 / VR
- 视频流（对话修改时实时预览）

### 3.5.2 技术实现（参照前文 KooEngine 拆解）
- 自研/集成实时光追引擎 + GPU 集群 + GPU 虚拟化多租户。
- 智能图形优化（降采样/缓存复用）保实时性。
- 流式传输闭环：用户输入→云端渲染→视频编码→终端解码。
- 终端零安装，手机/网页均可。

### 3.5.3 与对话联动
每次方案 JSON 变更 → 触发增量重渲染（仅重渲变化区域）→ 秒级刷新 3D Viewer。

---

## 3.6 工程 Agent：从方案到可施工交付

### 3.6.1 这是"目前没有、价值最高"的模块
把"好看方案"转成"能施工、能买、能装"的交付包：

```
design JSON
  → ① 拆单/算量：按构件库提取尺寸/数量/工艺
  → ② BOM 物料清单：主材/辅材/五金，带品牌/型号/单价
  → ③ 报价单：人工+材料+管理费，可分房间/分项目
  → ④ 施工图（CAD/DWG + PDF）：
       平面布置图 / 水电点位图 / 立面图 / 节点大样
  → ⑤ 施工说明：工序、注意事项、验收标准
```

### 3.6.2 技术实现
- 参数化构件库每个构件绑定：`工艺参数 + BOM 模板 + 单价 + 出图规则`。
- 规则引擎按户型+设计自动生成水电点位（基于规范与人体工学）。
- 输出标准 CAD（DXF/DWG），兼容工长现有流程。

### 3.6.3 价值
- 小白可直接拿图纸找工长，不被信息差宰。
- 方案即导购：BOM 直接对接电商/品牌，一键加购。

## 3.7 对话 Agent：自然语言 ↔ 方案 JSON

### 3.7.1 角色
用户用自然语言/发图提出修改，Agent 解析为对方案 JSON 的**结构化编辑指令**，触发全链路重算。

### 3.7.2 技术实现（LLM + Function Calling）
```
用户："主卧衣柜加长 30cm，参考这张图换个暖色调"
  → LLM 意图识别 + 槽位填充
  → 调用 Function：
       edit_furniture(id="主卧衣柜", delta_size=[300,0,0])
       apply_style(image_ref=上传图, scope="全局")
  → 校验（约束引擎）：加长后是否碰门/超墙？
       · 通过 → 提交 JSON 变更
       · 冲突 → 返回可选方案（"只能加 20cm，或移至西墙"）
  → 触发：重渲染 + 重算量 + 重出图 + 刷新 UI
```

### 3.7.3 关键能力
- **上下文记忆**：多轮对话保持方案状态。
- **冲突协商**：不能执行时给替代方案，而非简单拒绝。
- **可解释**：每次修改给出"改了什么、为什么、影响什么（价格/尺寸变化）"。
- **撤销/版本**：方案快照，可回退。

### 3.7.4 参考图对话
"把客厅换成这张图的风格" → 风格向量提取 → 应用到对应房间 → 重渲染。

# 4. 对话优化闭环（端到端流程）

```
[用户发图/说话]
      │
      ▼
对话 Agent → 意图解析 → Function Call
      │
      ├─→ 约束引擎校验（可否执行？）
      │        │
      │        ├─ 否 → 返回替代方案（协商）
      │        └─ 是 ↓
      │
      ├─→ 修改 方案 JSON（Single Source of Truth）
      │
      ├─→ 并行触发：
      │     ├─ 渲染层：增量重渲染 3D
      │     ├─ 工程 Agent：重算 BOM/报价/施工图
      │     └─ 规划 Agent：如需，重排布局
      │
      ▼
[统一刷新：3D Viewer + 图纸 + 报价 + 尺寸标注]
      │
      ▼
[用户继续对话 / 导出交付包]
```

**核心指标**：单次修改到全链路刷新 < 3 秒（渲染流 + 算量异步）。

# 5. 数据模型与资产库

## 5.1 参数化构件库（最关键资产）
每个构件包含：
```json
{
  "id": "wd_01", "cat": "wardrobe",
  "name": "现代简约趟门衣柜",
  "size": [1800,600,2400],        // 默认尺寸（可参数缩放）
  "param_rules": {                 // 参数约束
     "width": [900, 3000], "depth": [500, 650]
  },
  "materials": ["oak","white_eco"],
  "price": 3200,                   // 基准单价
  "bom_template": [...],           // 物料清单模板
  "drawing_rule": "...",           // 出图规则
  "style_tags": ["modern_minimal"]
}
```

## 5.2 约束规则库
以结构化规则文件管理（见 3.2），支持热更新、分城市/分规范版本。

## 5.3 风格向量库
预计算主流风格 CLIP embedding，支持参考图实时提取与匹配。

## 5.4 用户数据
户型、方案历史、会话、偏好、导出记录，用于个性化与持续优化。

# 6. 技术选型建议

| 模块 | 推荐技术 | 说明 |
|------|----------|------|
| 户型识别 | CNN/Transformer 分割 + OCR + 微分渲染 | 自研或基于开源（如 FloorSG） |
| LLM（规划/对话） | 私有化开源模型（Qwen/DeepSeek）+ 云 API 兜底 | 数据隐私 + 成本 |
| 风格提取 | CLIP / 扩散图像编码器 | 参考图向量化 |
| 生成式 3D | 3DGS / 扩散（Tripo/混元3D 类） | MVP 后引入 |
| 云渲染 | 自研实时光追 + GPU 集群（参考 KooEngine） | 或集成商用（如 RaysEngine） |
| 约束引擎 | 规则引擎（Drools/自研 DSL） | 领域知识沉淀 |
| 前端 3D | Three.js / WebGPU + 流式视频 | 多端统一 |
| 后端 | 微服务 + 消息队列（异步重算） | 高并发 |
| 存储 | 对象存储（模型/图）+ 关系/文档库（方案 JSON） | — |

# 7. MVP 范围与路线图

## 7.1 MVP（4–6 个月，验证核心价值）
- [ ] 户型图上传 → 识别（墙/门/窗/尺寸），人工校正 UI
- [ ] 约束引擎 V1（承重墙红线 + 基础人体工学）
- [ ] 规划 Agent（LLM 空间分配）
- [ ] 生成 Agent 路线 A（构件拼装，可控可施工）
- [ ] 云渲染 3D 效果 + 720° 全景
- [ ] 工程 Agent V1（BOM + 报价 + 平面布置图）
- [ ] 对话 Agent V1（尺寸/家具增删改 + 风格切换）
- [ ] 交付：3D + 尺寸 + 图纸 + 报价一键导出

## 7.3 远期
- 施工队对接、进度管理
- 智能家居联动设计
- 区域规范自动适配（跨城）
- 数字孪生住宅

# 8. 关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 承重墙误判导致安全隐患 | 极高 | 保守标记 + 强制人工确认 + 免责声明 |
| 生成方案不可施工 | 高 | 约束引擎前置 + 工程 Agent 校验 |
| 对话修改冲突无解 | 中 | 协商式替代方案 + 可视化选择 |
| 渲染成本过高 | 中 | GPU 虚拟化 + 智能优化 + 增量渲染 |
| 数据隐私（户型=家庭信息） | 中 | 私有化部署 + 脱敏 |
| LLM 幻觉（错误尺寸/价格） | 中 | 所有数值走规则引擎/BOM，不依赖 LLM 编造 |
| 构件库覆盖不足 | 中 | 众包 + 品牌合作 + UGC |

**最致命风险**：LLM 编造尺寸/价格。对策——**LLM 只做"意图与布局推理"，所有具体数值必须由构件库/规则引擎/BOM 计算，LLM 不生成最终数字。**

# 10. 与现有产品的差异化总结

| 维度 | 酷家乐/三维家 | 本方案（AI Agent） |

# 11. 结论

本方案的技术创新不在于某个新算法，而在于**把成熟单点技术集成为一个"约束感知、全链路同步、对话驱动"的室内设计 Agent 系统**。

最大壁垒是**约束规则库 + 单一数据骨架（方案 JSON）+ 对话闭环的工程实现**，而非 AI 模型本身。

MVP 可在 4–6 个月内以"可控构件拼装 + 云渲染 + 工程出图 + 对话修改"验证核心价值，率先实现"小白上传户型图，说话就能拿到可施工全套方案"的差异化体验。
