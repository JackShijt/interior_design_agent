# 逐功能深度拆解

## F1 户型图识别
**功能**：上传照片/扫描/CAD → 墙、门、窗、尺寸、房间名 → 结构化户型 JSON。

**技术难点**
1. **多源异构**：拍照畸变/阴影/反光 vs CAD 线稿 vs 老房手绘，风格差异巨大。
2. **线稿提取**：墙线粗细不一、断裂、双层线（内外墙），需骨架化+拓扑闭合。
3. **尺寸精度**：OCR 识别数字易错，比例校准依赖至少一个可靠标注。
4. **房间语义**：图标/文字歧义（"书房？储藏室？"）。
5. **算力**：海量用户并发上传，CPU 密集（见前文 kam 服务）。

**突破方向**
- 透视校正 + 多图融合预处理。
- 分割网络（墙/门窗/房间）+ 检测网络（符号）+ OCR 三路并行。
- 微分渲染闭环：渲染几何回图比对，反向优化参数（CVPR2021 思路）。
- 比例校准：优先用标注尺寸；缺失则用标准模数（如开间 3000/3300）推算并提示。
- 人工校正 UI 兜底 + 反馈回流训练。

**MVP 取舍**：先支持清晰 CAD/扫描图，拍照识别做降噪降级。

## F2 承重墙判定（最高危）
**功能**：识别/标记承重墙、梁、管井，禁止 AI 擅自拆改。

**技术难点**
1. **图上无显式标注**：很多户型图不标承重墙，纯靠视觉推断不可靠。
2. **误判代价极高**：拆错承重墙 = 结构安全事故。
3. **不规范图纸**：老房、自画图无标准。

**突破方向**
- **保守策略**：无法 100% 确定时，一律标记为"疑似承重"，默认禁止拆改。
- **多信号融合**：墙厚（≥240 多为承重）、墙体连续性、与梁关系、户型对称性辅助判断。
- **强制人工确认**：涉及承重墙改动，弹窗确认 + 免责声明，不自动执行。
- **外部数据**：对接房产/物业结构图（远期）。

**底线原则**：安全优先级 > 用户体验 > 自动化程度。

## F3 自然语言需求理解
**功能**："现代简约、两口之家、多收纳" → 空间分配策略。

**技术难点**
1. **口语模糊**：用户不会说"主卧衣柜靠东墙"，只说"想要大点衣柜"。
2. **隐含约束**："有娃"隐含圆角家具、插座防护。
3. **多意图交织**：一句话混风格+功能+预算。

**突破方向**
- LLM + 领域 Prompt + Few-shot 示例，做意图分类与槽位填充。
- 结构化快捷选项（风格/家庭/预算）降低自然语言负担。
- 规划 Agent 把模糊需求翻译成"房间功能 + 家具清单草案"，再交生成 Agent。
- 不确定时反问（"收纳优先级高，是否牺牲部分活动空间？"）。

---

## F4 约束感知生成（核心价值难点）
**功能**：生成既好看、又能施工的方案。

**技术难点**
1. **美观 vs 合规冲突**：AI 想放中岛，但厨房太小放不下。
2. **自由生成不可控**：扩散/3DGS 生成的结果常无法编辑、无精确尺寸。
3. **布局合理性**：动线、采光、家具间距需符合人体工学。

**突破方向**
- **路线 A（MVP 首选）可控构件拼装**：从参数化库选件 + 几何约束求解（靠墙/对齐/间距），100% 可施工。
- **约束引擎前置**：生成前裁剪不合规组合，生成中做约束优化。
- **路线 B（进阶）生成式 3D**：生成视觉 → 反投影到构件库"贴合"，保留可编辑性。
- 规划 Agent 先定空间策略，生成 Agent 在其约束内发挥。

---

## F5 参考图风格迁移
**功能**：发一张喜欢的图，把自家换成类似风格。

**技术难点**
1. **风格解耦**：图里混了"布局+材质+色调+造型"，需只提取风格、不照搬布局。
2. **适配自家约束**：参考图是 30㎡豪宅，自家 8㎡，直接套会翻车。
3. **材质可行**：参考图材质可能无对应可购商品。

**突破方向**
- CLIP / 扩散编码器提取风格向量（色调/材质/造型语言）。
- 风格仅作用于材质/色调层，布局仍由自家户型 + 约束引擎决定。
- 构件库做风格标签匹配，确保迁移后每件家具都有可购对应。

---

## F6 云渲染实时呈现
**功能**：3D 效果、720°、对话时实时刷新。

**技术难点**
1. **低延迟**：对话改一下，3D 要秒级刷新，不能等离线渲染。
2. **高画质 vs 成本**：光追画质高但贵，需平衡。
3. **并发**：海量用户同时渲染，GPU 资源调度。

**突破方向**
- GPU 集群 + 虚拟化多租户（参考 KooEngine 拆解）。
- **增量渲染**：只重渲变化区域（如只改衣柜，不重渲全屋）。
- 智能图形优化（降采样/缓存复用/重要性采样）。
- 流式传输闭环 + 边缘计算压延迟。

---

## F7 对话式修改
**功能**：说话即改方案（"衣柜加长 30cm"）。

**技术难点**
1. **精准编辑**：自然语言 → 对方案 JSON 的精确变更，不能"大概其"。
2. **冲突协商**：加长后碰门了怎么办？不能简单拒绝。
3. **多轮上下文**：连续对话要记住之前改了啥。
4. **可解释**：用户要知道改了什么、影响什么（价格/尺寸）。

**突破方向**
- LLM Function Calling：把口语映射为预定义函数（edit_furniture / apply_style）。
- 冲突时返回替代方案（"只能加 20cm，或移至西墙"）而非拒绝。
- 方案快照做撤销/重做 + 上下文记忆。
- 每次修改回显"变更 diff + 价格/尺寸变化"。

---

## F8 全链路同步（架构级难点）
**功能**：改一处，3D/尺寸/图纸/报价同步更新。

**技术难点**
1. **数据一致性**：多个视图（3D/2D/报价）不能各算各的，否则对不上。
2. **性能**：同步重算不能卡（用户等 >3s 就流失）。
3. **异步编排**：渲染慢、算量快，要并行不互等。

**突破方向**
- **单一数据骨架**：所有视图由同一份方案 JSON 驱动（本方案第一原则）。
- 事件驱动：JSON 变更 → 发消息 → 各下游订阅重算（渲染/算量/出图并行）。
- 渲染走流式（异步），算量/尺寸同步返回，UI 先更新轻量部分。

---

## F9 工程出图与报价
**功能**：方案 → BOM + 报价 + 施工图（CAD）。

**技术难点**
1. **构件→工艺映射**：每件家具/硬装要知道怎么施工、用什么料。
2. **水电点位自动生成**：需符合规范与人体工学，非简单摆放。
3. **报价准确**：材料单价、损耗、人工浮动。
4. **出图标准**：工长能看懂的 CAD，非装饰性图。

**突破方向**
- 构件库每个件绑定：`工艺参数 + BOM 模板 + 单价 + 出图规则`（资产壁垒）。
- 规则引擎按户型+设计自动算水电点位（基于规范/人体工学）。
- 输出标准 DXF/DWG，兼容现有工长流程。
- 报价分房间/分项，可下钻。

---

## F10 合规与安全（底线难点）
**功能**：不生成违规/危险方案。

**技术难点**
1. **规范多样**：分城市、分类型（住宅/商铺）规范不同。
2. **LLM 幻觉**：可能编造合规结论。
3. **责任边界**：AI 出的图真拿去施工，出问题谁负责。

**突破方向**
- 约束引擎红线（A 类结构安全）绝对阻断，不依赖 LLM 判断。
- 规范分 scope 加载（分城市/类型）。
- **免责声明 + 人工确认**：关键决策留人审。
- LLM 不生成最终数字/合规结论，只做推理，数值走规则引擎。

# 结论

- **最致命的不是算法，是"约束 + 数据骨架 + 闭环"的工程整合**——F2/F10/F8 是生死线。
- **MVP 应优先攻克 P0 + 部分 P1**，用可控构件拼装（F4 路线 A）绕过自由生成的不可控。
- **资产壁垒在构件库 + 约束规则库**（F4/F9），这是对手难抄的部分。
