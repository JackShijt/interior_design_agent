# 户型图识别方案选型（MVP）

> 目标：上传户型图 → 结构化 house JSON（walls / rooms / openings / 尺寸 mm）。
> 约束：单人团队、低成本、精度够用即可（MVP 放宽：墙体召回≥85%，尺寸误差≤8%）。

## 三方案对比

| 方案 | 精度 | 成本 | 周期 | 单人可行性 |
|------|------|------|------|-----------|
| (a) 现成 API/SaaS | 高且稳 | 按张计费，需联网 | 1–2 天接入 | ★★★★★ 最推荐 |
| (b) 开源模型 FloorSG 类 | 中（依赖图质量） | 免费，需自部署 GPU | 2–4 周调参 | ★★ 需 CV 功底 |
| (c) 自研 CV | 不稳定 | 时间极高 | 数月 | ☆ 不推荐 |

## MVP 推荐：(a) 现成 API 优先，预留 (b) 降级

理由：
1. 单人团队资源有限，先把"识别→生成→对话→出图"业务闭环跑通，识别准确性交给成熟服务；
2. 成本可控（按张计费，MVP 阶段量级极小）；
3. 接口统一为 `recognize(image_path) -> house`，后续可平滑替换为自部署模型而不动上游。

落地骨架见 `agents/understand.py`：默认尝试调用外部识别服务（由环境变量 `RECOGNITION_API_URL` / `RECOGNITION_API_KEY` 注入），失败或未配置时回退到内置 Mock 户型，保证开发期闭环不中断。

## 验收（MVP 放宽）
- 清晰 CAD 图：墙体召回≥85%，尺寸误差≤8%；
- 承重墙判定沿用保守策略，不确定标 `suspect_load_bearing`；
- 输出结构可被 generator 直接消费。
