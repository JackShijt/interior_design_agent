# AI 室内设计 Agent —— 研发任务拆分

> 配套文档：产品技术方案 / MVP补充(PRD+规则库+代码) / 功能与技术难点拆解
> 目的：把方案直接拆成研发可认领的任务、里程碑、依赖与验收
> 时间：2026-07-14
> 假设团队：后端×3、算法×2、前端×2、产品/设计×1（约 8 人，MVP 周期 4–6 月）

---

# 0. 任务拆分总原则

1. **以"单一数据骨架（方案 JSON）"为轴心**，所有任务最终都读写它。
2. **P0 生死线优先**：承重墙判定(F2)、合规安全(F10)、全链路同步(F8) 最先启动。
3. **MVP 用可控构件拼装(F4 路线A)**，不碰自由生成式 3D，降低不可控风险。
4. **每个功能点(F1~F10)映射一个 Epic**，难点映射为技术 Spike 或子任务。
5. **可运行原型先行**：先用 MVP补充文档 的代码框架跑通闭环，再逐步替换 Mock。

# 1. 里程碑（Milestone）规划

| 里程碑 | 时间 | 目标 | 关键交付 |
|--------|------|------|----------|
| M0 原型验证 | 第 1–4 周 | 跑通"JSON→约束→对话→算量"闭环 | 可运行 Demo（Flask） |
| M1 识别可用 | 第 5–9 周 | 户型图识别 + 承重墙标记 | 识别服务 + 校正 UI |
| M2 生成可施工 | 第 8–14 周 | 约束生成 + 3D 呈现 + 出图报价 | 生成+渲染+工程三服务 |
| M3 对话闭环 | 第 12–18 周 | 自然语言改方案 + 全链路同步 | 对话 Agent + 事件总线 |
| M4 MVP 发布 | 第 18–24 周 | 端到端可用 + 导出交付包 | 小程序上线 + 验收 |

> 注意 M1/M2/M3 有重叠（并行推进），M0 在最前。

# 2. Epic 与任务分解

## Epic-0：基础设施与数据骨架（P0，第 1–3 周）
**目标**：建立全系统的单一数据源与基础服务。

| 任务ID | 任务 | 负责 | 产出 | 依赖 |
|--------|------|------|------|------|

| T0.3 | 构件库数据模型 + 初始 50 件（床/柜/桌/沙发） | 后端+产品 | components.json | T0.1 |
| T0.4 | 约束规则库框架（YAML DSL + 加载器） | 算法 | constraints.yaml + loader | — |

| 任务ID | 任务 | 负责 | 产出 | 依赖 |
|--------|------|------|------|------|

| T1.2 | 图像预处理（透视校正/去噪/多图融合） | 算法 | 预处理服务 | T1.1 |
| T1.3 | 墙线分割 + 矢量化 + 拓扑闭合 | 算法 | 识别模型 v1 | T1.1 |
| T1.4 | 门窗检测 + OCR 尺寸识别 | 算法 | — | T1.3 |
| T1.5 | 比例校准（标注尺寸→毫米） | 算法 | — | T1.4 |
| T1.6 | 识别服务 API + 人工校正 UI | 后端+前端 | 上传/识别/校正页 | T0.2,T1.3 |

**目标**：标记承重墙/管井，禁止 AI 拆改，强制人工确认。

| 任务ID | 任务 | 负责 | 产出 | 依赖 |
|--------|------|------|------|------|
| T2.1 | 承重墙多信号融合判定（墙厚/连续性/梁关系） | 算法 | 判定模型 | T1.3 |
| T2.2 | "疑似承重"保守标记策略 + 置信度 | 算法 | — | T2.1 |
| T2.3 | 约束引擎 A 类红线规则（拆承重/开大洞/移管井） | 算法 | 规则集 | T0.4 |
| T2.4 | 强制人工确认弹窗 + 免责声明 | 前端+产品 | UI 组件 | T2.3 |
| T2.5 | 安全审计日志（谁确认了拆改） | 后端 | 日志 | T0.2 |

**难点**：T2.1 无标注推断不可靠 → 对策 T2.2 保守+人工。

# 3. 任务依赖关系图（简版）

```
T0(骨架) ─┬─→ T1(识别) ─→ T2(承重墙/P0)
          ├─→ T3(规划生成) ─→ T4(渲染) ─┐
          ├─→ T5(工程出图) ─────────────┤
          └─→ T6(对话同步/P0) ←──────────┘
                   ↑依赖 T3/T4/T5 事件接入
T7(风格) 依赖 T3
T8(发布) 依赖 T4/T5/T6
```

**关键路径**：T0 → T3 → T6（对话闭环是 MVP 核心价值，最晚但也最关键）。
**P0 并行**：T2（安全）应与 T0 同期启动，不阻塞其他。

# 5. 验收标准（MVP 出口）

| 维度 | 标准 |
|------|------|
| 识别 | 墙体 F1≥0.9，尺寸误差≤5% |
| 生成 | 10s 内出首版，约束通过率≥95% |
| 安全 | 承重墙拆改 100% 阻断/确认 |
| 渲染 | 首渲<10s，对话刷新<3s |
| 对话 | 修改全链路同步<3s，冲突有替代 |
| 交付 | 导出 PDF 含效果+尺寸+图纸+报价 |
| 体验 | 小白 3 步内出首版占比≥70% |

# 6. 风险任务预警（Spike 优先）

# 结论

本拆分把方案落为 **8 个 Epic、约 40 个任务**，以方案 JSON 为轴、P0（安全+同步）优先、MVP 用可控构件拼装。
每个 F 映射 Epic，每个难点映射 Spike/子任务，可直接导入项目管理工具（Jira/飞书）作为迭代 backlog。

> 与前述文档关系：产品技术方案（架构）→ MVP补充（PRD/规则/代码）→ 功能难点拆解（F1~F10）→ 本拆分（任务级）。
